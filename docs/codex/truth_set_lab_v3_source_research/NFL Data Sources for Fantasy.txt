Player‑level stats from the nflverse
nflreadr::load_player_stats() (R package) or nflreadpy.load_player_stats() (Python port) with summary_level="week"
Returns a tibble/data frame with ~115 variables (see data dictionary). Important fields include player_id (GSIS ID), season, week, season_type, team, opponent_team, position, completions, attempts, passing_yards, passing_tds, passing_interceptions, carries, rushing_yards, rushing_tds, targets, receptions, receiving_yards, receiving_tds, rushing_first_downs, receiving_first_downs, etc[1]. Format can be chosen (rds, csv, parquet, qs) via file_type argument.
Built from play‑by‑play data going back to 1999 and updated nightly during the season. The nflverse data schedule states that player stats are computed on the same nightly schedule as play‑by‑play data[2]. Setting seasons = TRUE loads all available seasons; specifying a numeric vector (e.g., 2010:2025) restricts the range.
player_id is the NFL GSIS identifier (used throughout nflverse)[3]. Additional keys: nflverse_player_id and play_id are not used here.
Comprehensive weekly box‑score data. Each row represents a player‑week (one player in one NFL game). The dataset includes box‑score stats (passing, rushing, receiving, defensive, special teams) plus derived metrics (EPA, fantasy_points, air yards, etc.). Because this dataset is derived from official NFL stat crews via play‑by‑play, it aligns closely with box‑score totals. rushing_first_downs and receiving_first_downs are included[1] and do not need to be derived manually.
Key fields to import for weekly fantasy modelling
Identifiers: player_id (GSIS), season, week, season_type, team and optionally opponent_team and position.
Passing production: completions, attempts, passing_yards, passing_tds, passing_interceptions[4].
Rushing production: carries, rushing_yards, rushing_tds, rushing_first_downs[5].
Receiving production: targets, receptions, receiving_yards, receiving_tds, receiving_first_downs[6].
Games/starts: Not included in this dataset. These can be approximated by counting the number of player‑week rows (games played) and, if needed, by joining depth charts (nflreadr::load_depth_charts()) or snap counts (nflreadr::load_snap_counts()) to determine starters (rows where depth_team / position_depth equals 1) and number of snaps played. The user should be aware that “games started” is not provided directly.
Additional metrics: rushing_fumbles, receiving_fumbles, passing_air_yards, receiving_air_yards, fantasy_points (standard scoring), fantasy_points_ppr, etc., are available for advanced modelling.
How to access the weekly data
R – call:
library(nflreadr)weekly_stats <- load_player_stats(seasons = TRUE, summary_level = "week", file_type = "csv")
This downloads a CSV file directly from the nflverse GitHub release and caches it locally. The file_type can be set to parquet for efficient reading in Python or rds for R.
Python – using the nflreadpy package:
import nflreadpyweekly_stats = nflreadpy.load_player_stats(seasons="all", summary_level="week", file_type="parquet")
Direct download – the underlying file can be downloaded from the nflverse GitHub release (e.g., https://github.com/nflverse/nflverse-data/releases/download/stats_player/player_stats_weekly.csv) and read with pandas or similar. Rely on nflreadr::csv_from_url() if you prefer not to install the package.

Season‑level player stats from the nflverse
nflreadr::load_player_stats() or nflreadpy.load_player_stats() with summary_level="reg" (regular season), "post" (post‑season) or "reg+post" (combined). This function aggregates the weekly box‑score stats into season totals.
Same data frame structure as the weekly version, but aggregated by player and season. Columns include season, team, position, completions, attempts, passing_yards, passing_tds, passing_interceptions, carries, rushing_yards, rushing_tds, targets, receptions, receiving_yards, receiving_tds, rushing_first_downs, receiving_first_downs, fantasy points, etc.
Available for all seasons covered by the nflverse play‑by‑play data (1999–present) and updated nightly during the season[2].
player_id (GSIS) identifies the player. Because the aggregated dataset sums over games and possibly over multiple teams (in the case of traded players), there may be multiple rows per player and season if a player switched teams; the team column reflects the last team for which the player accrued stats.
Preferred for season totals. Using summary_level="reg" ensures that only regular‑season games are included. summary_level="post" isolates playoff production, and "reg+post" combines both. rushing_first_downs and receiving_first_downs are aggregated sums[1]. Games and starts are not explicitly included; they can be computed by counting weeks or by joining depth charts/snap counts.
3 Best source for rushing/receiving first downs
The player‑level stats dataset described above already contains rushing_first_downs and receiving_first_downs for each player‑week and for each season.
The data dictionary explicitly lists “rushing_first_downs – First downs on rush attempts (incl. scrambles)” and “receiving_first_downs – total number of first downs gained on receptions”[1].
These fields are derived from play‑by‑play by nflverse and do not need to be recomputed manually. The aggregate_game_stats.R function in the nflfastR package calculates them by summing plays where first_down_rush or first_down_pass is TRUE[7][8].
For modelling, simply import these columns. If additional derived metrics (e.g., rushing_first_down_rate) are needed, they can be computed locally (e.g., rushing_first_downs / carries).

5 Mapping nflverse players to Sleeper IDs
The nflreadr::load_ff_playerids() function provides a cross‑reference between nflverse IDs and fantasy platform IDs. The data dictionary lists gsis_id (the nflverse player_id) and sleeper_id among many other platform identifiers[27]. To map players:
Load the mapping table:
ids <- nflreadr::load_ff_playerids()
Join your weekly/season stats table to ids on player_id = gsis_id to attach sleeper_id, fantasypros_id, espn_id, etc. For example:
stats_with_ids <- dplyr::left_join(weekly_stats, ids, by = c("player_id" = "gsis_id"))
When exporting to Sleeper, use the sleeper_id field. Note that DynastyProcess (which maintains the mapping) updates the table weekly[28].
6 Implementation notes for local import
Package installation: Use install.packages("nflreadr") in R or pip install nflreadpy in Python to access the data without manual downloads.
File type: For large historical datasets, parquet or rds files are more efficient. Use file_type = "parquet" in load_player_stats() to directly download a Parquet file, which can be read by pandas/pyarrow or arrow in R.
Caching: nflreadr caches data in a local directory (default .cache/nflreadr). You can set a custom cache with options(nflreadr.cache_path = ...) to avoid repeated downloads.
Combining seasons: Pass seasons = TRUE or a vector of seasons to load_player_stats(); do not loop over years manually. The function returns a combined data frame.
Database storage: For long‑term use, consider loading the Parquet files into a database (e.g., DuckDB). Example in Python:
import pandas as pdimport duckdbweekly_stats = nflreadpy.load_player_stats(seasons="all", summary_level="week", file_type="parquet")con = duckdb.connect('nfl_stats.duckdb')con.execute("CREATE TABLE player_week AS SELECT * FROM weekly_stats")
Handling missing values: Stats will be 0 or NA when a player did not record a specific stat in a game. Use fillna(0) or dplyr::mutate(across(where(is.numeric), ~replace_na(.x, 0))) to standardize.
Games/starts: If needed, compute games as n_distinct(week) per player and starts by joining depth charts or snap counts. Depth chart updates have changed after 2024 (timestamped rather than weekly)[29], so users must choose appropriate timestamps.
7 Gaps and considerations
Games and starts: The nflverse player stats tables do not include explicit counts for games played or games started. These fields must be derived from weekly appearances or by joining snap counts and depth charts. Depth charts changed to timestamped updates after the 2024 season[29], making it harder to infer weekly starters.
Data completeness: The nflverse play‑by‑play and player stats cover regular seasons and post‑seasons back to 1999. Data before 1999 is not available. Snap counts are only available from 2012 onwards (Pro Football Reference) and may be missing for some early seasons.
Injury/participation data: The participation data source changed and does not update during the season after 2024[30]. If you need in‑game participation (e.g., to identify starters), this may be incomplete for 2025–26.

Use of this data for modelling and research is generally allowed but commercial redistribution may be restricted by the original data providers (NFL, PFR). Refer to the nflverse licence and the terms of service for PFR and Sleeper. Always attribute nflverse when publishing results.
Confidence score: The proposed solution relies on official nflverse data and documentation as of May 15 2026. Because nflverse pipelines are actively maintained, fields and update schedules may evolve. Cross‑checking the data dictionary and update schedule provides high confidence (~85 %) that load_player_stats() is the correct source for weekly and seasonal production data. However, deriving games and starts may require additional data sources or manual logic, so there is some uncertainty (~15 %).
