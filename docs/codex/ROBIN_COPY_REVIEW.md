# Robin Copy Review

## Verdict
YELLOW

## One-Sentence Read
The voice is mostly disciplined and mission-fit, but the visible copy still needs sharper analytical humility and any "Not found" surface must be treated as a copy and trust failure.

## Mission Voice Fit
The language matches the project mission when it stays close to concrete command-center nouns: roster, official top five, forced release, keeper pressure, pick value, model audit, import errors, and local snapshot. The best copy feels like a private analytical tool for one owner, not a fantasy advice brand. The weaker copy appears when labels drift toward recommendation certainty, broad "war room" drama, or repair/process language that belongs in internal docs rather than the working app.

## Delicate Wording Risks
- "Recommendations" can imply advice rather than deterministic model output. Prefer "model label", "score label", or "review flag" where the app is showing formula output.
- "Which players should be kept, dropped, or shopped?" is useful in mission copy, but app copy should soften this into "Keep/drop/shop flags" or "Model label" so it does not overclaim.
- "Trade options can save value" risks sounding certain. Prefer "Trade options that may preserve value under current assumptions."
- "Players are likely to enter the draft pool" should show source and confidence. Prefer "Projected draft-pool candidates" with "why" and "source" visible or nearby.
- Any visible "Not found" page is a severe hospitality failure for an analytical tool: it tells the user nothing about what broke, what route they wanted, or how to recover.
- Internal repair terms such as "handoff", "workflow", "polish", "manager-ready", "ready for service", or "repair active pack" should stay out of the app surface.
- "War Room Rank" is acceptable as product language, but it should be paired with a plain definition in the audit surface so it does not sound like a hidden guru score.
- Labels like "OFFER", "DECLINE", "AVOID", and "POLITICAL RISK" are crisp, but they should remain table labels with model inputs visible nearby, not standalone directives.

## Beautiful Language Opportunities
- Rename any visible error state from "Not found" to a useful local-app recovery line with route, expected page, and next action.
- Add small source/date language beside tables: "Snapshot: 2026 pre-declaration" or "Source: active local data pack."
- Make model surfaces more humble with column labels like "Computed score", "Formula inputs", "Assumption", "Confidence", and "Why."
- Use "forced-release pressure" consistently instead of switching between more dramatic or advisory phrasing.
- Keep page headers short: "Import Review", "Niners Roster", "War Board", "Trade Central", "Draft Room", "League Intel", "Model Audit."
- Replace broad explanatory paragraphs with compact table captions that answer: what is this table, what data powers it, and what decision it supports.
- Use "review" language for human judgment: "Review before acting", "Manual check needed", "Low confidence", "Missing official rank."

## Voice Rules
- Keep visible app copy short, concrete, and table-first.
- Separate computed output from advice: say "score", "flag", "label", "projection", "assumption", and "confidence."
- Do not imply prediction certainty. Use "projected", "under current assumptions", or "model output" where needed.
- Keep internal build language out of the product UI.
- Prefer football and league nouns over generic product nouns.
- Do not explain the whole system on the first screen.
- Put formulas, caveats, and long explanations behind Model Audit, expanders, or detail panels.
- Keep Official Rank, Market Rank, War Room Rank, and My Rank visibly separate.
